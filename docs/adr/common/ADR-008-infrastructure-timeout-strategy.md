# ADR-COMMON-008: 인프라 방어 전략으로서의 타임아웃(Timeout) 도입

## 상태
승인됨 (Accepted)

## 맥락 (Context)
포인트 충전 API는 외부 결제 시스템(Mock PG)과 통신합니다. 외부 시스템은 우리 서버의 통제 범위를 벗어나 있으며, 지연(Latency)이나 장애가 발생할 가능성이 상존합니다. 모놀리식 아키텍처 환경에서는 하나의 외부 호출 지연이 서버의 가용 스레드를 모두 점유하여, 결제와 무관한 메뉴 조회나 매장 찾기 등 전체 시스템의 마비로 이어지는 연쇄 장애(Cascading Failure)를 유발할 수 있습니다. 이를 방지하기 위한 인프라 방어막 설계가 필요합니다.

## 결정 (Decision)
1. **서킷 브레이커 도입 보류 및 타임아웃 채택**:
    - MSA 환경이 아니며 현재 학습 단계임을 고려하여, 상태 관리가 복잡한 서킷 브레이커(Resilience4j 등) 대신 직관적이고 즉각적인 효과를 내는 '타임아웃' 전략을 우선 적용합니다.
2. **RestTemplate 전역 타임아웃 설정**:
    - **Connect Timeout**: 3초
    - **Read Timeout**: 5초
3. **RestConfig 클래스 생성**: `RestTemplateBuilder`를 통해 타임아웃이 적용된 `RestTemplate` 빈(Bean)을 설정하여 외부 API 호출 시 표준으로 사용합니다.

## 논의 과정 (Discussion)
*   **Developer**: "MSA가 아닌데도 서킷 브레이커 같은 기술이 필요한가요?"
*   **Architect**: "모놀리식은 모든 기능이 하나의 프로세스에서 돌아가므로, 외부 PG 지연 시 전체 기능이 같이 죽을 위험이 더 큽니다. 장애 격리가 필수적입니다."
*   **Developer**: "서킷 브레이커는 너무 생소합니다. 더 단순한 대안은 없나요?"
*   **Architect**: "타임아웃과 벌크헤드(격리)가 있습니다. 논의 결과, 학습 효율과 프로젝트 정체성 사이의 균형을 위해 가장 직관적인 '타임아웃'만 가볍게 적용하기로 결정했습니다."
*   **User**: "타임아웃 설정을 하되, 왜 3초와 5초인지 명확한 근거를 제시해 주세요."

## 설정 근거 (Rationale)
1. **Connect Timeout (3초)**:
    - **근거**: TCP 핸드셰이크를 포함한 네트워크 연결 수립 시간입니다. 일반적으로 서버 간 통신은 1초 이내에 완료되나, 일시적인 네트워크 순단이나 라우팅 이슈를 고려하여 3번의 재시도(Retransmission) 기회를 가질 수 있는 충분한 시간인 3초를 부여했습니다. 3초가 지나도 연결이 안 된다면 해당 서버는 장애 상태로 판단하는 것이 안전합니다.
2. **Read Timeout (5초)**:
    - **근거**: 외부 결제 시스템이 비즈니스 로직을 처리하고 응답을 보내주는 시간입니다. 실제 카드사/PG사 승인 프로세스는 대개 2~3초 내외에 종료됩니다. 사용자 경험(UX) 측면에서 모바일 앱 사용자가 '모래시계'를 견딜 수 있는 심리적 한계선인 5초를 설정했습니다. 5초가 넘어가면 요청을 강제 종료하여 우리 서버의 스레드 점유를 해제하고, 사용자에게 빠른 에러 응답을 주는 것이 시스템 전체 가용성에 유리합니다.

## 결과 (Consequences)
- **장점**: 외부 장애로 인한 전체 시스템 마비 방지. 코드 및 설정의 단순함 유지.
- **단점**: 외부 서버가 지속적인 장애 상태일 경우, 매 요청마다 타임아웃 시간만큼의 스레드 대기가 발생함 (서킷 브레이커만큼의 완벽한 차단은 아님).
