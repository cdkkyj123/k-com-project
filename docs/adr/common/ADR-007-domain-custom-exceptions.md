# ADR-COMMON-007: 도메인별 커스텀 예외 도입 및 에러 처리 구조 개편

## 상태
승인됨 (Accepted)

## 맥락 (Context)
초기 프로젝트에는 시스템의 비즈니스 도메인과 무관한 수많은 에러 코드가 혼재되어 있었으며, 모든 예외를 공통 `ServiceException` 하나로 처리하고 있었습니다. 이는 에러의 발생 근원(도메인)을 명확히 식별하기 어렵게 만들고, 특정 도메인에 특화된 예외 처리 로직을 확장하는 데 한계가 있었습니다.

## 결정 (Decision)
1. **도메인 전용 예외 클래스 도입**:
    - `ServiceException`을 상속받는 `UserException`, `MenuException`, `StoreException`, `PointException`, `OrderException`을 생성합니다.
    - 각 도메인의 서비스와 엔티티는 반드시 자신의 도메인 예외를 던지도록 리팩토링합니다.
2. **ErrorCode 정비 및 슬림화**:
    - 현재 스마트 오더 프로젝트의 도메인과 무관한 레거시 에러 코드들을 전면 삭제합니다.
    - 도메인별로 필요한 에러 코드를 그룹화하여 재구성함으로써 코드의 응집도를 높입니다.
3. **객체지향적 예외 발생**:
    - 엔티티 내부에서 상태 검증 시 적절한 도메인 예외를 직접 발생시킴으로써 도메인 모델의 자율성을 보장합니다.

## 논의 과정 (Discussion)
*   **Developer**: "단순히 `ServiceException`에 `ErrorCode`만 갈아 끼워도 동작하는데, 왜 별도의 클래스를 만드나요?"
*   **Architect**: "로그 분석이나 `@ExceptionHandler`에서 도메인별로 서로 다른 처리를 해야 할 때, 타입 기반으로 분기하는 것이 훨씬 명확하고 확장성이 좋기 때문입니다."
*   **Developer**: "기존의 불필요한 에러 코드들은 어떻게 처리할까요?"
*   **Architect**: "우리 프로젝트의 정체성을 해치는 '죽은 코드'입니다. 과감히 삭제하여 비즈니스 로직에만 집중할 수 있는 환경을 만듭니다."

## 결과 (Consequences)
- **장점**: 에러의 출처가 명확해짐. 도메인별 예외 처리 전략(Retry 여부 등)을 다르게 가져가기 용이함.
- **단점**: 클래스 파일 수가 늘어남.
