# ADR-010: 엔티티 변경에 따른 정교한 캐시 무효화 전략

## 상태
승인됨 (Accepted)

## 맥락 (Context)
현재 매장 및 메뉴 정보는 성능 향상을 위해 Redis에 캐싱되고 있습니다. 그러나 메뉴의 가격이 바뀌거나 매장의 영업 상태가 변경(`OPEN` -> `CLOSED`)되었을 때, 캐시가 제때 갱신되지 않으면 사용자가 주문할 수 없는 매장을 보거나 틀린 가격으로 주문하는 데이터 불일치 문제가 발생합니다. 전체 캐시를 한꺼번에 날리는 것은 성능에 치명적이므로, 변경된 데이터와 연관된 캐시만 선택적으로 제거하는 전략이 필요합니다.

## 결정 (Decision)
1. **필드/상태 기반 캐시 이빅션(Cache Evict)**:
    - 메뉴 품절(`SOLD_OUT`)이나 매장 상태 변경 시, 관련 서비스 메서드에 `@CacheEvict`를 적용합니다.
    - `@CacheEvict(value = "menuList", allEntries = true)` 등을 활용하여 특정 도메인의 캐시 뭉치만 효과적으로 무효화합니다.
2. **이벤트 기반 캐시 갱신**:
    - 중요한 정보 변경 시 Spring Event나 Kafka 이벤트를 발행하고, 이를 구독하는 리스너에서 연관된 여러 캐시 키를 일관성 있게 삭제합니다.

## 논의 과정 (Discussion)
*   **Developer**: "캐시 만료 시간(TTL)만으로 충분하지 않을까요?"
*   **Architect**: "재고가 떨어지거나 매장이 문을 닫는 정보는 '즉시성'이 중요합니다. TTL에만 의존하면 10분의 TTL 동안 사용자는 계속 잘못된 정보를 보게 됩니다. 쓰기(Write) 작업 시 명시적으로 캐시를 날려주는 것이 정합성 측면에서 필수적입니다."
*   **Developer**: "캐시 스탬피드 방지(PER 알고리즘 등)도 같이 적용해야 할까요?"
*   **Architect**: "캐시 스탬피드는 트래픽이 극도로 몰리는 상황을 가정한 복잡한 로직입니다. 현재는 정밀한 '무효화' 자체에 집중하고, 성능 최적화는 넥스트 스텝으로 미룹니다."

## 결과 (Consequences)
- **장점**: 데이터 변경 사항이 사용자에게 실시간으로 정확히 반영됨. 불필요한 DB 조회를 줄이면서도 정합성 유지.
- **단점**: 데이터 수정 로직마다 캐시 삭제 코드가 포함되어 코드 복잡도가 약간 증가함.
