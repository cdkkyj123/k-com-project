# ADR-002: Kafka 토픽 및 컨슈머 전략 (파티셔닝 및 다중 컨슈머 그룹)

## 상태
승인됨 (Accepted)

## 맥락 (Context)
실시간 주문 데이터를 수집 플랫폼으로 전송할 때, 데이터의 양이 늘어남에 따라 병렬 처리 능력이 요구됩니다. 또한, 동일한 메시지를 서로 다른 목적으로 사용하는 여러 시스템(예: 데이터 수집, 실시간 알림 등)이 존재할 수 있으므로 독립적인 컨슈머 그룹 구성이 필요합니다.

## 결정 (Decision)
1.  **토픽 파티셔닝**:
    - `coffee-orders` 토픽의 파티션 수를 3개로 설정합니다 (브로커 수와 동일하게 맞춤).
    - 이를 통해 최대 3개의 컨슈머가 병렬로 메시지를 처리할 수 있도록 합니다.
2.  **다중 컨슈머 그룹**:
    - `data-platform-group`: 주문 내역을 데이터 수집 플랫폼으로 전송하는 주 목적의 컨슈머 그룹.
    - `analytics-group`: 실시간 통계 분석을 위한 별도의 컨슈머 그룹 (향후 확장 고려).
3.  **애플리케이션 설정 업데이트**:
    - `application.yaml`에서 클러스터의 모든 브로커(`localhost:9092,9094,9095`)를 Bootstrap Servers로 설정합니다.
    - 토픽 생성 시 `partitions: 3`, `replication-factor: 3`을 명시적으로 설정하는 Bean을 추가합니다.

## 결과 (Consequences)
- **장점**: 수평적 확장이 용이해지며, 특정 컨슈머 그룹의 장애가 다른 그룹에 영향을 주지 않음.
- **단점**: 파티션 수 증가에 따른 순서 보장 문제(동일 키 사용으로 해결 가능) 및 리소스 관리 복잡도 증가.
