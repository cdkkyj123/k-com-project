# [ADR-012] 레디스 인프라 고도화 전략: 클러스터 도입 검토 및 의사결정

## 1. 개요
- **상태**: 보류/기각 (Deferred/Rejected)
- **날짜**: 2026-05-08

## 2. 배경 (Context)
현재 시스템은 단일 Redis 노드를 사용하여 분산 락, 캐시, 멱등성 체크를 수행하고 있습니다. 시스템의 성장에 대비하여 카프카와 같은 클러스터 구조로 레디스를 고도화할 때의 실익을 검토하였습니다.

## 3. 검토 내용 (Investigation)

### 3.1 도입 시 장점 (Pros)
- **고가용성 (High Availability)**: 마스터 노드 장애 시 슬레이브가 즉시 승격(Failover)되어 서비스 중단을 방지함.
- **수평적 확장 (Scalability)**: 수천 건의 동시 락 요청이 여러 노드에 분산되어 단일 노드의 CPU/메모리 부하를 경감함.

### 3.2 도입 시 단점 및 제약 사항 (Cons)
- **운영 리소스 증가**: 최소 6개 노드(3 Master + 3 Slave)가 필요하여 인프라 비용 및 도커 리소스 사용량이 급증함.
- **멀티 키(Multi-key) 제약**: 여러 메뉴에 대해 한 번에 락을 잡을 때, 키들이 서로 다른 해시 슬롯에 배정되면 원자적 처리가 불가능함. 이를 해결하기 위해 `{hash-tag}`를 강제하면 특정 노드에 부하가 쏠리는 'Hot-spot' 이슈 발생 가능.
- **데이터 정합성 리스크**: 복제 방식이 비동기이므로, 마스터가 락을 잡고 죽었을 때 슬레이브에 락 정보가 전달되지 않은 상태로 승격되면 락 유실(Lock Loss)이 발생할 미세한 확률이 존재함.
- **설계 복잡도**: 클라이언트(Redisson)의 토폴로지 스캔 설정 및 네트워크 오버헤드 관리 필요.

## 4. 의사결정 (Decision)
**단일 노드(Single Node) 체제를 유지**하기로 결정하였습니다.

## 5. 결정 근거 (Rationale)
1. **비용 대비 효율**: 현재 트래픽 규모에서 클러스터의 확장성은 과잉 스펙(Over-engineering)임.
2. **정합성 우선**: 우리 시스템은 다중 메뉴 주문(OrderItem) 시 여러 키에 락을 걸어야 하는데, 클러스터 환경의 해시 슬롯 제약은 비즈니스 로직의 복잡도를 불필요하게 높임.
3. **충분한 성능**: 현재 Facade 패턴을 통해 락과 트랜잭션의 경계를 분리함으로써 단일 노드에서도 충분한 정합성과 성능을 확보함.

## 6. 향후 과제 (Future Work)
- 초당 수천 건 이상의 요청이 발생하는 시점에 **Redis Sentinel** 또는 **Redis Cluster** 도입을 재검토함.
- 도입 시에는 반드시 `{Hash Tag}` 설계안을 포함하여 멀티 키 연산의 원자성을 보장해야 함.
