# ADR-004: 컨트롤러 레이어 응답 래핑 및 ResponseEntity 표준화

## 상태
승인됨 (Accepted)

## 맥락 (Context)
비즈니스 로직을 담당하는 서비스 레이어는 순수한 데이터(DTO)만을 반환하고, 클라이언트에게 전달할 최종 응답 규격(HTTP 상태 코드 및 공통 응답 포맷)은 컨트롤러 레이어에서 결정하는 것이 관심사 분리 관점에서 적절합니다. 이를 통해 서비스 레이어의 재사용성을 높이고 컨트롤러에서 응답에 대한 책임을 명확히 합니다.

## 결정 (Decision)
1.  **서비스 레이어 순수 데이터 반환**:
    - 모든 서비스 메서드는 `ApiResponseDto`로 래핑하지 않고, 순수한 데이터 객체(DTO)를 반환합니다.
2.  **컨트롤러 레이어 응답 래핑**:
    - 컨트롤러는 서비스로부터 받은 데이터를 `ApiResponseDto.success()`로 래핑합니다.
    - 최종적으로 `ResponseEntity.ok()` (또는 적절한 상태 코드 메서드)를 사용하여 반환합니다.
    - 예시: `return ResponseEntity.ok(ApiResponseDto.success(service.method()));`

## 결과 (Consequences)
- **장점**: 서비스 레이어가 웹 계층의 응답 규격에 의존하지 않아 테스트와 재사용이 용이해집니다. 컨트롤러에서 표준화된 방식으로 응답을 구성할 수 있습니다.
- **단점**: 컨트롤러 코드가 약간 더 길어질 수 있으나, 가독성과 역할 분담 면에서 이득이 큽니다.
