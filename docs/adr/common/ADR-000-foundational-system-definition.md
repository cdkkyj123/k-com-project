# ADR-000: 시스템 근간 정의 및 아키텍처 수립의 여정

## 상태
승인됨 (Accepted)

## 개요 (Overview)
본 문서는 "단순한 커피 주문 앱"에서 출발하여 "전국구 규모의 실시간 스마트 오더 시스템"으로 비즈니스 모델과 기술 아키텍처를 확립해 나간 **사고의 과정과 의사결정 이력**을 상세히 기록합니다.

### 🎯 우리가 만드는 시스템은 무엇인가?
본 시스템은 **스타벅스의 '사이렌 오더'나 투썸플레이스의 '투썸 하트'** 와 같은 **전국 규모 프랜차이즈의 모바일 주문 및 실시간 매장 관제 시스템**을 모델로 합니다.

해당 서비스에서 **모바일 주문 시 선택한 매장으로 주문이 들어가고** 이를 **이용자가 픽업 할 수 있는 서비스**를 목표로 합니다. 

단순히 DB에 데이터를 넣고 빼는 수준을 넘어, 다음과 같은 핵심 비즈니스 프로세스를 수행합니다.
1.  **중앙 오더링**: 전국의 수천만 사용자가 모바일 앱을 통해 주문 및 포인트 결제를 수행합니다.
2.  **실시간 매장 전파**: 중앙에서 처리된 주문은 즉시 **Kafka** 메시징 시스템을 타고 각 매장의 **POS(판매시점관리) 단말기**로 전송되어 제조를 시작하게 합니다.
3.  **라이브 데이터 대시보드**: 전국 매장에서 쏟아지는 방대한 주문 스트림을 실시간으로 수집·가공하여, 사용자에게는 "전국 인기 메뉴 TOP 3"를, 본사에는 "실시간 매출 현황"을 초 단위로 제공합니다.

## 논의 단계 및 의사결정 프로세스

### 1단계: 프로젝트의 정체성 확립 (Generic vs. Specialized)
*   **논의 제안**: 프로젝트 초기, 시스템의 구체적인 사용 환경(키오스크, POS, 웹사이트 등)에 대한 정의가 부족했음.
*   **질문 (Developer)**: "단순 조회 중심의 POS 시스템(B)으로 갈 것인가, 성능과 확장성이 중요한 대규모 모바일 주문 시스템(A)으로 갈 것인가?"
*   **판단 및 결정 (User)**: **"대규모 처리 역량 증명을 위해 모바일 주문 시스템(A) 방향을 채택한다."**
*   **인사이트**: 이 결정을 통해 "단순 기능 구현"이 아닌 "고성능/고가용성 아키텍처"라는 프로젝트의 대전제가 확립됨.

### 2단계: 현실적 복잡도와 기술적 본질 사이의 균형 (Geospatial vs. Scale)
*   **논의 제안**: 모바일 주문 시스템이라면 매장별 상태 관리와 사용자의 위치 기반 매장 검색이 필요함.
*   **대안 제시 (Developer)**: Redis GeoHash나 DB Spatial Index를 이용한 실시간 위치 기반 매장 매칭 API 도입 제안.
*   **비판적 피드백 (User)**: "위치 기반 로직은 기술적 복잡도를 높여 '대규모 데이터 처리'라는 본질을 흐릴 수 있다. 또한 매장별로 데이터를 단순히 쪼개는 것이 과연 '대규모'라는 명칭에 걸맞은가?"
*   **인사이트**: 기술적 화려함(위치 계산)보다 데이터의 흐름과 규모(Scale) 자체에 집중해야 한다는 핵심 방향성을 재확인함.

### 3단계: "대규모"의 재정의 (Distributed Sources to Centralized Aggregation)
*   **문제 의식**: 매장별로 데이터가 분산되면 각 접점의 트래픽은 낮아질 텐데, 이를 어떻게 대규모 시스템이라 부를 수 있는가?
*   **해결 논리 도출 (Synthesis)**:
    1.  **데이터의 밀집**: 개별 매장은 소량일지라도, 전국 수천 개 매장에서 쏟아지는 주문은 중앙 서버와 Kafka 입장에서 **초당 수천 건의 거대한 스트림 데이터**가 됨.
    2.  **실시간 집계(Aggregation)**: 분산된 발생지 데이터를 한곳으로 모아 **"전국 실시간 인기 메뉴"**로 가공하는 과정이야말로 진정한 대규모 데이터 엔지니어링의 영역임.
*   **최종 결정**: **"전국 프랜차이즈 스마트 오더 시스템"** 시나리오 확정. 
    - Kafka는 매장 POS로의 주문 전달(내결함성)과 전국 데이터 수집(파이프라인)의 이중 역할을 수행.
    - Redis는 고속 포인트 결제(동시성)와 실시간 전국 랭킹 집계(성능)의 핵심 저장소로 활용.

### 4단계: 기술적 디테일의 완성 (Offset vs. Cursor)
*   **질문 (User)**: "성능을 위해 Offset 기반이 아닌 Cursor 기반 페이징이 더 도움이 되는가?"
*   **비교 분석 (Developer)**: Offset 방식의 성능 저하(뒤로 갈수록 느려짐)와 데이터 중복 문제를 설명하고, Cursor 방식의 O(1) 성능 우위와 인피니트 스크롤 UI 적합성을 강조.
*   **결정**: 스마트 오더의 특성상 특정 페이지 이동보다 지속적인 메뉴 탐색이 중요하므로 **Cursor 기반 페이징을 전면 채택**.

## 최종 결론: 왜 이 시스템인가?

우리는 단순히 코드를 짜는 것을 넘어, **"분산된 수많은 소스(매장 주문)로부터 유입되는 대량의 데이터를 실시간으로 수집하고, 이를 정합성 있게 가공하여 인사이트(인기 차트)를 제공한다"**는 전형적인 대규모 백엔드 모델을 구축하기로 했습니다.

이 과정에서 기술적 선택(Kafka, Redis, Cursor Paging, Outbox Pattern)은 단순히 유행을 따르는 것이 아니라, **비즈니스의 현실성과 규모의 확장성**을 해결하기 위한 필수적인 도구로서 그 정당성을 얻었습니다.
