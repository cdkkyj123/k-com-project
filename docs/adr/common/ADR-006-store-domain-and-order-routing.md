# ADR-006: 매장 도메인 도입 및 주문 라우팅 전략

## 상태
승인됨 (Accepted)

## 맥락 (Context)
스타벅스의 '사이렌 오더'와 같은 "전국 프랜차이즈 스마트 오더 시스템" 시나리오를 구현하기 위해서는 사용자가 주문을 보낼 특정 매장을 선택할 수 있어야 합니다. 또한, 중앙 서버에서 처리된 주문이 각 매장의 POS 시스템으로 정확히 전달(Routing)되기 위해서는 주문 데이터에 매장 식별자(`storeId`)가 반드시 포함되어야 합니다.

## 결정 (Decision)
1.  **Store 도메인 신설**: 매장의 정보(이름, 주소)와 영업 상태(OPEN, CLOSED)를 관리하는 독립적인 도메인을 생성합니다.
2.  **Order 엔티티 확장**: 주문 정보에 `storeId` 필드를 추가하여 어떤 매장에서 발생한 주문인지 기록합니다.
3.  **주문 검증 로직 추가**: 주문 생성 시 해당 매장이 존재하고 현재 '영업 중(OPEN)'인지 확인하는 유효성 검사 단계를 추가합니다.
4.  **Kafka 메시지 라우팅 기반 마련**: Outbox에 저장되는 주문 이벤트 페이로드에 `storeId`를 포함시켜, 향후 매장별 POS 컨슈머가 자신의 매장 주문만 필터링하여 처리할 수 있도록 설계합니다.

## 근거 (Rationale)
- **비즈니스 현실성**: 실제 프랜차이즈 환경에서는 매장별로 주문을 접수하고 제조하므로 매장 도메인 분리가 필수적입니다.
- **확장성**: 매장별로 주문 데이터를 분리함으로써 향후 특정 매장에 트래픽이 몰리는 'Hotspot' 문제를 Kafka 파티셔닝 등으로 해결할 수 있는 기반이 됩니다.
- **관심사 분리**: 매장 관리(Store)와 주문 처리(Order) 도메인을 분리하여 각 도메인의 책임을 명확히 합니다.

## 결과 (Consequences)
- **장점**: 현실적인 스마트 오더 흐름을 시스템에 투영할 수 있으며, 매장별 주문 관리 및 통계 산출이 용이해집니다.
- **단점**: 주문 생성 시 매장 상태 확인을 위한 추가적인 도메인 간 협력(Service Call)이 발생합니다.
