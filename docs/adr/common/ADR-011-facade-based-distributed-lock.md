# [ADR-011] Facade 패턴을 통한 분산 락과 트랜잭션 경계 분리

## 1. 개요
- **상태**: 채택됨 (Accepted)
- **날짜**: 2026-05-07

## 2. 배경 (Context)
이전 시도([ADR-011 v1] 비관적 락 계획)에서 언급했듯이, 분산 락(Redisson)과 트랜잭션(@Transactional)의 생명주기가 일치하지 않아 데이터 정합성 이슈가 발생했습니다.
초기에는 DB 비관적 락(`FOR UPDATE`)을 통해 정합성을 강제하려 했으나, 대규모 트래픽 환경에서 DB 커넥션 풀 고갈 및 성능 저하가 우려되었습니다.

## 3. 상세 분석: 레이스 컨디션의 근본 원인
Spring의 `@Transactional`은 프록시 방식으로 동작하며, 메서드가 리턴된 **후**에 커밋이 발생합니다.
반면 분산 락은 메서드 내부의 `finally` 블록에서 해제됩니다.
```
스레드 A: 락 획득 -> 비즈니스 로직(재고-1) -> 락 해제 -> (커밋 대기 중)
스레드 B:              (락 획득 성공) -> 비즈니스 로직(재고 조회: 여전히 과거값) -> 재고-1 -> 락 해제 -> 커밋
스레드 A:                                                   -> 커밋 완료
```
위와 같이 락이 풀리는 시점과 실제 데이터가 DB에 반영되는 시점 사이의 찰나의 간극(Gap)이 문제를 야기합니다.

## 4. 해결 방안: Facade 패턴 도입
락을 획득하고 해제하는 책임을 트랜잭션 범위 밖의 **Facade(퍼사드) 계층**으로 완전히 분리합니다.

### 4.1 변경된 실행 흐름 (TO-BE)
1.  **Facade**: 분산 락 획득 (Redisson)
2.  **Facade**: `@Transactional` 서비스 메서드 호출 (트랜잭션 시작)
3.  **Service**: 비즈니스 로직 수행 및 변경사항 반영
4.  **Service**: 메서드 종료 (트랜잭션 커밋 완료 및 데이터 DB 반영)
5.  **Facade**: 분산 락 해제 (정합성이 보장된 상태에서 다음 스레드에게 권한 양도)

## 5. 의사결정 (Decision)
- **Facade 계층 신설**: `OrderFacade`, `PointFacade`를 도입하여 락 관리를 전담합니다.
- **Service 리팩토링**: `OrderTransactionService` 등 트랜잭션 전용 서비스에서 Redisson 의존성을 제거하고 순수 로직만 유지합니다.

## 6. 기대 효과 (Consequences)
- **완벽한 정합성**: 데이터가 커밋된 후에만 락이 해제되므로 초과 판매나 갱신 유실이 원천 차단됩니다.
- **고성능 유지**: DB 비관적 락을 사용하지 않으므로 커넥션 풀 점유 시간이 짧아지고 대규모 부하 상황에서도 안정적인 처리가 가능합니다.
- **관심사 분리**: 락 제어(인프라)와 비즈니스 처리(도메인)의 책임이 명확히 분리됩니다.
