# ADR-009: Outbox 패턴 성능 최적화 (Batch 처리 및 DLQ 도입)

## 상태
승인됨 (Accepted)

## 맥락 (Context)
현재 `OutboxScheduler`는 DB에서 미발행 이벤트를 하나씩 조회하여 전송하는 단순 폴링 방식입니다. 트래픽이 전국구 단위로 급증할 경우, 수만 건의 이벤트를 개별적으로 처리하는 것은 스케줄러의 병목을 유발하고 Kafka 발행 지연으로 이어집니다. 또한, 특정 메시지가 반복적으로 실패할 경우 전체 전송 프로세스가 지연되거나 무한 재시도되는 문제를 방지해야 합니다.

## 결정 (Decision)
1. **이벤트 배치 전송(Batch Processing)**:
    - 스케줄러가 한 번 실행될 때 최대 100건(조절 가능)의 이벤트를 일괄 조회하여 Kafka로 전송합니다.
    - 이를 통해 DB 커넥션 횟수와 네트워크 오버헤드를 줄입니다.
2. **DLQ(Dead Letter Queue) 상태 도입**:
    - `MAX_RETRY_COUNT`(예: 5회)를 초과한 메시지는 상태를 `FAILED`가 아닌 `DLQ`로 변경합니다.
    - `DLQ` 상태의 메시지는 일반적인 스케줄러 조회 대상에서 제외하여 시스템 전체의 지연을 막고, 추후 관리자가 수동으로 분석/복구할 수 있도록 격리합니다.

## 논의 과정 (Discussion)
*   **Developer**: "배치 처리 중에 하나만 실패하면 어떻게 하나요?"
*   **Architect**: "배치 내 개별 메시지마다 try-catch를 적용하여 실패한 메시지만 `retryCount`를 올리고, 성공한 메시지는 즉시 `SENT`로 확정 짓는 '안전한 배치' 방식을 택합니다."
*   **Developer**: "DLQ에 쌓인 데이터는 어떻게 처리하나요?"
*   **Architect**: "별도의 관리자 화면이나 알림 시스템을 통해 DLQ 발생을 인지하고, 원인 분석 후 '재발행' 버튼을 통해 상태를 다시 `INIT`으로 돌려주는 프로세스를 권장합니다."

## 결과 (Consequences)
- **장점**: 대량의 주문 이벤트 발행 성능 대폭 향상. 장애 발생 시 영향도 격리.
- **단점**: 배치 크기 설정에 따른 튜닝 필요. DLQ 모니터링 공수 발생.
