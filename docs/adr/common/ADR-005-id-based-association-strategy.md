# ADR-005: ID 기반 엔티티 연관 관계 전략

## 상태
승인됨 (Accepted)

## 맥락 (Context)
JPA의 객체 참조를 이용한 직접적인 연관 관계 매핑(@ManyToOne, @OneToMany 등)은 객체 그래프 탐색을 편리하게 해주지만, 대규모 분산 시스템 및 도메인 주도 설계(DDD) 관점에서는 여러 문제를 야기할 수 있습니다. 특히 N+1 문제, 강한 결합도, 그리고 영속성 컨텍스트 관리의 복잡성 등이 주요 병목 지점이 될 수 있습니다.

## 결정 (Decision)
1.  **ID 참조 방식 채택**:
    - 엔티티 간의 관계를 객체 참조가 아닌 **단순 ID(Long)** 값을 저장하는 방식으로 관리합니다.
2.  **도메인 간 결합도 감소**:
    - 각 도메인(Order, User, Menu 등)은 서로의 내부 엔티티 구조를 알 필요 없이 ID 값으로만 참조하여 물리적/논리적 결합도를 낮춥니다.
    - 이는 향후 마이크로서비스 아키텍처(MSA)로의 전환을 용이하게 합니다.
3.  **N+1 문제의 원천 차단**:
    - 객체 참조가 없으므로 지연 로딩(Lazy Loading)으로 인한 예상치 못한 N+1 쿼리 발생 가능성을 설계 단계에서 제거합니다.
4.  **성능 최적화**:
    - 조인이 필요한 경우 서비스 계층에서 필요한 ID 목록을 수집하여 `IN` 절을 사용하는 쿼리(`findByIdIn`)로 데이터를 가져오거나, CQRS 패턴을 적용하여 조회 전용 모델을 구성합니다.

## 결과 (Consequences)
- **장점**: 
    - 도메인 간의 독립성이 보장되어 유지보수가 쉬워집니다.
    - JPA의 고질적인 성능 문제인 N+1을 근본적으로 방지할 수 있습니다.
    - 엔티티가 가벼워지며 데이터베이스 트랜잭션 범위 제어가 명확해집니다.
- **단점**: 
    - 객체 그래프 탐색(`order.getUser().getName()`)이 불가능하며, 필요한 데이터를 가져오기 위해 서비스 레이어에서 추가적인 조회가 필요할 수 있습니다.
    - 데이터 무결성 제약(FK)을 애플리케이션 레벨에서 관리해야 하는 부담이 증가할 수 있습니다.
