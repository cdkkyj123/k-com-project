# ADR-POINT-003: 외부 결제 시스템 연동 시뮬레이션 및 정합성 설계

## 상태
승인됨 (Accepted)

## 맥락 (Context)
포인트 충전을 위해서는 외부 결제 시스템(PG)과의 연동이 필수적입니다. 하지만 실제 PG사 API를 연동하는 것은 테스트 결제 환경 구축, API 키 관리, 실제 과금 발생 등 개발 외적인 번거로움이 큽니다. 본 프로젝트의 핵심 목표는 "대규모 트래픽 환경에서의 데이터 정합성 해결"이므로, PG사 SDK 자체보다는 외부 시스템 연동 시 발생하는 불확실성(네트워크 지연, 실패, 타임아웃)을 어떻게 처리할 것인지가 더 중요한 과제입니다.

## 결정 (Decision)
1. **Mock PG(가상 결제 시스템) 시뮬레이션**:
    - 실제 PG SDK를 연동하지 않고, 인터페이스를 통해 가상의 결제 서비스를 구현합니다.
    - 무작위 지연(Latency)과 간헐적 실패(Failure)를 발생시켜 실환경과 유사한 변동성을 시뮬레이션합니다.
2. **트랜잭셔널 아웃박스(Transactional Outbox) 패턴 적용**:
    - 결제 요청 이력을 DB에 먼저 저장(Pending 상태)하고, 외부 연동이 성공하면 상태를 업데이트합니다.
    - 시스템 장애로 인해 상태 업데이트가 누락되더라도, 스케줄러가 아웃박스를 조회하여 재시도하거나 보상 트랜잭션을 실행합니다.
3. **Webhook 처리 및 결과 정합성 보장**:
    - 외부 시스템으로부터의 비동기 알림(Webhook) 상황을 가정하여, 동기적 응답과 비동기 알림 중 먼저 도착하는 데이터를 기준으로 정합성을 맞춥니다.

## 논의 과정 (Discussion)
*   **Developer**: "포트원(PortOne)이나 토스페이먼츠 같은 실제 SDK를 붙이는 게 더 실무적이지 않을까요?"
*   **Architect**: "실제 SDK 연동은 API 명세서에 맞춰 호출하는 수준에 그칩니다. 우리의 도전 과제는 '외부 시스템이 응답하지 않거나 네트워크가 끊겼을 때 내 시스템의 데이터를 어떻게 보존하는가'입니다."
*   **Developer**: "지연과 실패를 어떻게 시뮬레이션하나요?"
*   **Architect**: "결제 요청 시 `Thread.sleep()`으로 지연을 주고, `Random` 값을 이용해 10% 정도의 에러를 발생시킬 것입니다. 이때 아웃박스 패턴이 제대로 동작하여 포인트 유실이 없는지 검증하는 것이 핵심입니다."
*   **Developer**: "결과적으로 외부 연동 로직보다는 '신뢰할 수 없는 외부 시스템과의 통신 전략'을 보여주는 데 집중하자는 것이군요."
*   **Architect**: "정확합니다. 그것이 대규모 분산 시스템을 다루는 백엔드 개발자의 핵심 역량입니다."

## 결과 (Consequences)
- **장점**: 외부 의존성 없이 독립적인 테스트 및 개발 가능. 데이터 정합성 보장 전략(Outbox)의 실효성 증명.
- **단점**: 실제 PG사의 특수한 에러 코드나 복잡한 보안 프로토콜을 경험하기 어려움.
