# ADR-POINT-001: 포인트 트랜잭션 동시성 제어 전략

## 상태
승인됨 (Accepted)

## 맥락 (Context)
스타벅스 사이렌 오더와 같은 대규모 시스템에서 포인트 충전 및 차감은 매우 빈번하게 발생합니다. 특히 동일 사용자가 여러 단말기에서 동시에 요청을 보내거나, 시스템 오류로 인한 중복 요청이 발생할 경우 데이터 정합성(Race Condition) 문제가 발생할 수 있습니다. 
단순한 DB 업데이트는 트래픽이 몰릴 때 정합성을 보장하기 어렵고, DB 비관적 락(`SELECT FOR UPDATE`)은 트래픽이 극도로 높은 상황에서 커넥션 점유 시간이 길어져 병목 현상을 유발할 수 있습니다.

## 결정 (Decision)
1.  **Redisson 분산 락(Distributed Lock) 도입**:
    - `userId`를 키로 하는 분산 락을 사용하여 특정 사용자의 포인트 처리를 직렬화합니다.
    - DB 락보다 가벼우며, Redis의 단일 스레드 특성을 이용하여 고성능의 락 관리가 가능합니다.
2.  **트래픽 격리**:
    - 포인트 처리는 독립적인 트래픽 키(User ID)를 가지므로, 전체 시스템이 아닌 특정 유저 단위로 락을 걸어 병목을 최소화합니다.
3.  **트랜잭션 경계 설정**:
    - 락을 획득한 후 트랜잭션을 시작하고, 트랜잭션이 완전히 커밋된 후 락을 해제하도록 구현하여 데이터 가시성(Visibility) 문제를 방지합니다.

## 논의 과정 (Discussion)
*   **Developer**: "DB 비관적 락으로도 충분하지 않을까요?"
*   **User**: "전국 규모의 시스템입니다. DB 커넥션 하나하나가 소중한 자원인데, 락 대기 시간 동안 커넥션을 잡고 있는 것은 위험합니다."
*   **Developer**: "그렇다면 Redis를 이용한 분산 락이 적합합니다. Redisson 라이브러리의 Pub/Sub 기반 락을 사용하면 스핀 락보다 CPU 부하가 적습니다."
*   **User**: "좋습니다. 다만 락 해제가 트랜잭션 커밋보다 먼저 일어나지 않도록 주의해주세요."

## 결과 (Consequences)
- **장점**: 고가용성 환경에서 안전한 포인트 정합성 보장. DB 부하 감소.
- **단점**: Redis 의존성 증가. 네트워크 지연 발생 가능성.
