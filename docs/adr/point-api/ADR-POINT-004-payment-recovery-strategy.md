# ADR-POINT-004: 결제 상태 보정 전략 (Payment Recovery Strategy)

## 상태 (Status)
제안됨 (Proposed)

## 배경 (Context)
포인트 충전 프로세스는 외부 결제 시스템과의 연동이 필수적입니다. 이 과정에서 네트워크 타임아웃, 외부 시스템의 일시적 장애, 클라이언트의 비정상 종료 등으로 인해 로컬 데이터베이스에는 `PENDING` 상태로 남았지만 실제 결제는 성공했을 가능성이 존재합니다. 
반대로 결제는 실패했지만 상태가 여전히 `PENDING`일 수도 있습니다. 이러한 데이터 불일치를 해결하고 포인트의 최종 정합성을 보장하기 위한 보정 메커니즘이 필요합니다.

## 결정 (Decision)
결제 상태의 최종 정합성을 보장하기 위해 **스케줄러(Scheduler)**와 **웹훅(Webhook)** 두 가지 방식을 병행하여 도입합니다.

### 1. 웹훅 (Webhook) 수신 API 도입
- 외부 결제 업체(PG/카드사 등)로부터 결제 결과를 비동기적으로 전달받는 엔드포인트를 제공합니다.
- 장점: 실시간에 가까운 상태 업데이트가 가능하며, 클라이언트가 응답을 받지 못한 경우에도 서버 간 통신으로 정합성을 맞출 수 있습니다.
- 고려사항: 멱등성(Idempotency) 보장이 필수적입니다. 이미 `SUCCESS`인 경우 중복 충전되지 않도록 처리해야 합니다.

### 2. 보정 스케줄러 (Recovery Scheduler) 구현
- 일정 시간(예: 5분) 이상 `PENDING` 상태로 머물러 있는 건들을 주기적으로 조회합니다.
- 외부 결제 시스템의 상태 조회 API를 호출하여 해당 건의 실제 성공 여부를 확인합니다.
- 성공인 경우 포인트를 충전하고 상태를 `SUCCESS`로 변경하며, 실패인 경우 `FAILED`로 변경합니다.
- `ShedLock`을 활용하여 분산 환경에서도 중복 실행을 방지합니다.

### 3. 예외 상황 처리
- 외부 조회 API조차 실패하는 경우, 다음 스케줄링 주기에서 재시도하도록 `PENDING` 상태를 유지합니다.
- 수동 개입이 필요한 극단적인 상황을 위해 로그 및 알림 시스템을 연계합니다.

## 결과 (Consequences)
- 외부 시스템 장애 시에도 포인트 충전의 누락을 최소화할 수 있습니다.
- 사용자에게 일관된 충전 경험을 제공하며 운영자의 수동 보정 리소스를 절감합니다.
- 데이터베이스 부하를 고려하여 스케줄링 주기와 조회 대상의 범위를 적절히 조절해야 합니다.
