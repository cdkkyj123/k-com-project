# ADR-POINT-002: 포인트 충전 멱등성 및 비즈니스 유효성 검증 전략

## 상태
승인됨 (Accepted)

## 맥락 (Context)
네트워크 불안정이나 클라이언트의 중복 클릭으로 인해 동일한 포인트 충전 요청이 여러 번 전달될 수 있습니다. 포인트는 금전적 가치를 지니므로 중복 처리는 시스템에 치명적인 오류를 야기합니다. 또한, 충전 금액의 범위나 사용자의 최대 보유 한도 등 비즈니스 규칙이 엄격히 준수되어야 하며, 이러한 검증 로직의 위치에 대한 설계가 필요합니다.

## 결정 (Decision)
1. **HTTP Header 기반 멱등성 키 도입**:
    - `Idempotency-Key`를 필수(`required = true`) 헤더로 지정합니다.
    - 클라이언트는 각 요청마다 고유한 UUID를 생성하여 전달해야 합니다.
2. **Redis를 활용한 중복 요청 방어**:
    - 성공적으로 처리된 `Idempotency-Key`를 Redis에 1시간 동안 저장합니다.
    - 동일한 키로 요청이 올 경우, 비즈니스 로직을 수행하지 않고 기존의 성공 응답을 반환하거나 중복 요청 에러를 반환합니다.
3. **도메인 엔티티(`User`) 중심의 유효성 검증**:
    - 최소/최대 충전 금액 제한, 최대 보유 한도 초과 여부 등의 검증 로직을 `User` 엔티티 내부에 구현합니다.
    - 서비스 레이어는 절차만 제어하고, 실제 데이터의 정합성과 규칙은 객체 스스로가 보호하도록 합니다.

## 논의 과정 (Discussion)
*   **Developer**: "멱등성 키를 서버에서 먼저 발급받고 충전을 진행하는 게 안전하지 않을까요?"
*   **Architect**: "서버 발급 방식은 추가적인 네트워크 왕복(Round-trip)이 발생합니다. 클라이언트가 UUID를 생성해 보내는 방식이 표준적인 API 설계에 가깝고 구현이 가볍습니다."
*   **Developer**: "중복 요청 방어 기간은 어느 정도가 적당할까요?"
*   **Architect**: "일반적인 네트워크 재시도나 사용자의 실수에 대응하기에는 1시간이면 충분합니다. 영구 보관은 Redis 메모리 부하를 초래할 수 있습니다."
*   **Developer**: "검증 로직을 `PointService`에 두면 가독성이 더 좋지 않을까요?"
*   **Architect**: "서비스 레이어에 로직이 몰리면 '뚱뚱한 서비스(Fat Service)'가 되고 엔티티는 '빈약한 도메인 모델(Anemic Domain Model)'이 됩니다. 포인트 보유량이라는 상태를 가진 `User` 엔티티가 스스로의 상태를 보호하는 것이 객체지향적입니다."

## 결과 (Consequences)
- **장점**: 중복 결제 사고 원천 차단. 엔티티 응집도 향상으로 유지보수 용이.
- **단점**: 클라이언트의 구현 복잡도 소폭 증가(UUID 관리). Redis 가용성에 대한 의존성.
