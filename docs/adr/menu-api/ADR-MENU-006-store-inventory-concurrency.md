# ADR-MENU-006: 매장별 메뉴 재고 관리 및 분산 락 동시성 제어

## 상태
승인됨 (Accepted)

## 맥락 (Context)
현재 시스템은 메뉴의 가격과 정보만 관리하고 있으며, 실제 매장의 재고 상황을 반영하지 못하고 있습니다. 대규모 프랜차이즈 시스템에서는 특정 매장의 인기 메뉴가 조기에 품절될 수 있으며, 주문 시 포인트 차감과 재고 차감이 동시에, 그리고 원자적으로 이루어져야 합니다. 재고 차감 과정에서 Race Condition이 발생할 경우 실제 재고보다 더 많은 주문이 들어오는 '초과 판매(Overselling)' 문제가 발생할 수 있습니다.

## 결정 (Decision)
1. **매장별 재고 엔티티(`MenuStock`) 신설**:
    - `Menu` 엔티티와 별개로, 각 매장(`storeId`)과 메뉴(`menuId`) 쌍에 대한 재고 수량(`quantity`)을 관리하는 엔티티를 생성합니다.
2. **이중 분산 락(Multi-Distributed Lock) 적용**:
    - 주문 시 `userId` 기반의 포인트 락과 `storeId:menuId` 기반의 재고 락을 모두 획득한 후 트랜잭션을 시작합니다.
    - Redisson을 사용하여 고성능의 락 관리를 수행합니다.
3. **자동 품절(Sold-out) 처리**:
    - 재고 차감 결과가 0이 되는 순간, 해당 매장의 해당 메뉴 상태를 `SOLD_OUT`으로 즉시 변경하고 캐시를 갱신합니다.

## 논의 과정 (Discussion)
*   **Developer**: "DB의 비관적 락으로 재고를 관리하는 것이 더 간단하지 않을까요?"
*   **Architect**: "전국 수천 개 매장에서 동시에 발생하는 재고 확인 요청을 DB 락으로 처리하면 커넥션 풀 고갈과 성능 저하가 우려됩니다. Redis 분산 락을 통해 DB 부하를 분산시키고 원자성을 보장하는 것이 확장성에 유리합니다."
*   **Developer**: "포인트 락과 재고 락을 동시에 잡으면 데드락(Deadlock) 위험은 없나요?"
*   **Architect**: "락을 획득하는 순서(예: 항상 User ID 락 -> Stock 락 순서)를 일관되게 유지하고, 타임아웃을 짧게 설정하여 데드락 상황을 방지합니다."

## 결과 (Consequences)
- **장점**: 정확한 재고 관리 및 초과 판매 방지. 매장별 특수 상황 반영 가능.
- **단점**: 락 관리 포인트 증가로 인한 로직 복잡도 상승. Redis 의존성 강화.
