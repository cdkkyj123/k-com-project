# ADR-MENU-005: 최근 7일 인기 메뉴 실시간 집계 및 슬라이딩 윈도우 전략

## 상태
승인됨 (Accepted)

## 맥락 (Context)
"최근 7일간 인기 있는 메뉴 3개 조회" 요구사항을 달성하기 위해, 기존의 무거운 DB 집계 방식에서 벗어나 Kafka와 Redis를 활용한 실시간 스트리밍 집계 아키텍처를 설계함. 이 과정에서 데이터 정합성, 메모리 효율성, 초기화 방식에 대한 심도 있는 기술 논의가 진행됨.

## 논의 과정 및 의사결정 이력

### 1단계: 실시간 집계 모델 제안 (Developer)
- **제안**: 주문이 발생할 때마다 Kafka를 통해 이벤트를 발행하고, 컨슈머가 Redis의 Sorted Set(ZSET) 점수를 올리는 방식 제안.
- **근거**: 조회 시마다 발생하는 DB `GROUP BY` 부하를 제거하고 O(1) 속도로 실시간 랭킹을 제공하기 위함.

### 2단계: 데이터 초기화 및 보존 전략에 대한 의문 (User)
- **질문**: "요구사항에 따라 7일에 한 번씩 데이터를 초기화해야 하는가? 그리고 Redis 데이터를 DB에 다시 쓰는(Write-back) 작업이 필요한가?"
- **분석 및 답변 (Developer)**: 
    - **하드 리셋의 문제**: 7일마다 통째로 지우면 리셋 직후 데이터 공백이 발생함.
    - **슬라이딩 윈도우 도입**: '일자별 키 분리(Daily Rolling Key)' 전략 제안. 매일 새로운 키에 점수를 쌓고, 조회 시 최근 7일치 키를 합산함.
    - **Write-back의 가치**: Redis는 휘발성이므로 통계 히스토리 보존과 복구를 위해 일자별 합계 데이터를 DB에 기록하는 것이 비즈니스 인텔리전스(BI) 측면에서 유리함.

### 3단계: Redis 메모리 부하에 대한 우려 (User)
- **의문**: "주문마다 갱신되는 데이터를 7일 동안 Redis에 들고 있으면 메모리를 너무 많이 차지하지 않는가?"
- **기술적 해명 (Developer)**:
    - **데이터 경량성**: ZSET에는 오직 `메뉴ID(Long)`와 `점수(Double)`만 저장됨. 메뉴가 500개라면 하루치 데이터는 약 50KB 수준으로 매우 미미함.
    - **규모의 한계**: 사용자가 수백만 명으로 늘어나도 저장되는 엔트리 수는 '메뉴 가짓수'에 고정되므로 메모리 사용량이 우상향하지 않음.
    - **자동 최적화**: 각 일자별 키에 **8일의 TTL(만료 시간)**을 설정하여 Redis가 오래된 데이터를 자동으로 삭제하도록 설계함.

---

## 결정 (Decision)
1.  **데이터 구조**: Redis Sorted Set을 사용하여 `popular_menus:yyyyMMdd` 형식의 일자별 키를 운용함.
2.  **집계 방식**: Kafka Consumer가 주문 이벤트 수신 시 현재 날짜 키의 점수를 1 증가시킴.
3.  **조회 방식**: `ZUNIONSTORE` 명령(또는 RedisTemplate의 union)을 통해 최근 7일간의 키를 합산하여 상위 3개를 추출함.
4.  **자원 관리**: 각 일자별 키에 8일의 TTL을 부여하여 메모리 사용량을 일정하게 유지함.

## 결과 (Consequences)
- **장점**: 
    - 데이터 공백 없는 정확한 '최근 7일' 통계 제공.
    - 극도로 낮은 메모리 점유율과 초고속 응답 성능.
    - DB 부하의 획기적인 감소.
- **단점**: 일자별 키를 관리하고 합산하는 로직의 복잡도 증가.
