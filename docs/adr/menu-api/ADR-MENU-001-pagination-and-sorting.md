# ADR-MENU-001: 페이징 전략 전환 (Offset to Cursor-based)

## 상태
승인됨 (Accepted)

## 맥락 (Context)
초기 설계에서는 Spring Data JPA의 표준 방식인 **Offset 기반 페이징**을 사용하여 메뉴 목록을 반환하였습니다. 하지만 시스템의 방향성이 "대규모 트래픽을 처리하는 전국구 스마트 오더 시스템"으로 정의됨에 따라, 수천 개의 메뉴와 방대한 주문 히스토리를 효율적으로 처리하기 위한 고성능 페이징 전략이 요구되었습니다.

## 논의 과정 및 대안

### 1. 최초 제안: Offset 기반 페이징 유지
- **내용**: `page`와 `size` 파라미터를 사용하여 DB의 `OFFSET` 기능을 활용.
- **장점**: 구현이 매우 단순하고 특정 페이지 번호로의 직접 이동이 가능함.
- **거절 사유**: 데이터 양이 늘어날수록 성능이 저하되는 치명적인 단점이 있음 (예: 100만 번째 데이터 조회 시 앞선 100만 건을 모두 읽어야 함). 또한 페이징 도중 데이터 삽입/삭제 시 중복 노출 문제가 발생함.

### 2. 중간 논의: 위치 기반 매장 검색 및 분산 전략
- **내용**: 대규모 데이터 처리를 위해 사용자의 실시간 위경도를 계산하여 가장 가까운 매장을 매칭하는 로직 제안.
- **사용자 피드백**: 위치 기반 로직은 기술적 복잡도와 도메인 지식 요구량이 너무 높아 "대규모 데이터 처리"라는 본질적인 학습 목표를 흐릴 수 있음을 지적함. 또한, 단순히 매장별로 데이터를 쪼개는 것만으로는 "대규모"라는 명명에 걸맞은 난이도를 증명하기 부족할 수 있다는 우려를 표함.

### 3. 최종 채택안: 전국 단위 스마트 오더 시나리오 기반 Cursor 페이징
- **내용**: 특정 페이지 번호가 아닌, 마지막으로 조회된 데이터의 ID(`cursor`)를 기준으로 다음 데이터를 가져오는 **No-Offset 방식** 채택.
- **채택 이유**:
    - **성능적 우위**: 인덱스를 직접 탐색하므로 데이터 규모와 관계없이 조회 속도가 O(1)에 수렴함.
    - **비즈니스 정합성**: 모바일 인피니트 스크롤 UI에 최적화되어 있으며, 실시간으로 메뉴 상태가 변하는 스마트 오더 환경에서 데이터 중복 누락 문제를 해결함.
    - **논리적 타당성**: 사용자 피드백을 수용하여, 위치 기반의 복잡함은 덜어내되 "전국 수천 개 매장의 주문을 중앙에서 스트리밍 집계한다"는 시나리오를 확립함으로써 대규모 처리의 명분을 확보함.

## 결정 (Decision)
- 메뉴 목록 및 각종 이력 조회 API에서 Offset 방식을 폐기하고 **Cursor 기반(No-Offset) 페이징**을 전면 도입함.
- QueryDSL을 사용하여 마지막 PK 값을 기준으로 `where id > :lastId` 조건을 추가하고 `limit`을 통해 데이터를 추출함.

## 결과 (Consequences)
- **장점**: 억 단위의 데이터에서도 일정한 조회 성능 보장, 사용자 경험(UX) 개선, 데이터 정합성 확보.
- **단점**: 특정 페이지(예: 10페이지)로 바로 이동하는 기능 구현이 어려움 (단, 본 시스템의 모바일 UI 특성상 불필요한 기능으로 판단됨).
