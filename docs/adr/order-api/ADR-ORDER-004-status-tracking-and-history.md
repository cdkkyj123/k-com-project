# ADR-ORDER-004: 주문 상태 생명주기 세분화 및 이력 조회 API

## 상태
승인됨 (Accepted)

## 맥락 (Context)
현재 주문 상태는 단순 성공/실패 위주로 구성되어 있어, 사용자가 매장에서 자신의 커피가 제조 중인지, 혹은 픽업 가능한 상태인지 실시간으로 파악하기 어렵습니다. 또한, 사용자가 과거에 주문했던 내역을 다양한 조건(날짜, 상태 등)으로 효율적으로 검색할 수 있는 기능이 부재합니다.

## 결정 (Decision)
1. **주문 상태(OrderStatus) 세분화**:
    - `PREPARING`(제조 중), `READY`(픽업 대기), `PICKED_UP`(수령 완료) 상태를 추가하여 실제 매장 운영 흐름을 반영합니다.
2. **QueryDSL 기반의 고성능 이력 조회**:
    - `OrderQueryRepository`를 신설하고 QueryDSL을 사용하여 기간 필터링, 상태 필터링이 포함된 커서 기반 페이징 API를 구현합니다.
3. **상태 변경 이벤트 확장성 확보**:
    - 상태 변경 메서드를 캡슐화하여, 향후 상태 변경 시 알림(SSE, Push)을 보낼 수 있는 확장 지점(Hook)을 마련합니다.

## 논의 과정 (Discussion)
*   **Developer**: "조회 API에서 Offset 페이징을 쓰면 안 되나요?"
*   **Architect**: "주문 내역은 계속해서 쌓이는 데이터입니다. 뒤로 갈수록 느려지는 Offset 방식 대신, 대규모 데이터 처리에 최적화된 Cursor 기반 페이징을 ADR-000의 원칙에 따라 고수합니다."
*   **Developer**: "상태가 너무 많아지면 관리가 힘들지 않을까요?"
*   **Architect**: "각 상태는 비즈니스 로직(환불 가능 여부 등)의 기준이 되므로, 명확하게 세분화하는 것이 향후 정책 확장에 더 유리합니다."

## 결과 (Consequences)
- **장점**: 사용자 편의성(UX) 향상. 복잡한 검색 조건에서도 빠른 응답 속도 보장.
- **단점**: 주문 상태 전이에 따른 비즈니스 규칙 정의 및 검증 로직 추가 필요.
