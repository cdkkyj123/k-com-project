# ADR-ORDER-002: 다중 메뉴 주문 구조 개편 (장바구니 도입)

## 상태
승인됨 (Accepted)

## 맥락 (Context)
초기 주문 시스템은 단일 메뉴 주문(`menuId` 직접 참조)만 가능했습니다. 그러나 실제 스타벅스 사이렌 오더와 같은 서비스에서는 한 번의 주문으로 여러 종류의 메뉴를 다양한 수량으로 주문할 수 있어야 합니다. 이를 위해 데이터 모델을 확장하고 관련 비즈니스 로직을 리팩토링할 필요가 있습니다.

## 결정 (Decision)
1. **주문 및 주문 항목 분리**:
    - `Order` 엔티티에서 `menuId`, `price`를 제거하고, 전체 주문 금액을 나타내는 `totalPrice`를 추가합니다.
    - 주문 내 개별 메뉴와 수량을 관리하는 `OrderItem` 엔티티를 신설합니다.
2. **ID 기반 연관 관계 유지**:
    - 프로젝트의 핵심 원칙(ADR-005)에 따라, `Order`와 `OrderItem` 간에 JPA의 객체 그래프 참조(`@OneToMany`) 대신 `orderId`를 통한 단방향 ID 참조를 유지합니다.
3. **주문 상태 도입**:
    - 주문의 생명주기 관리를 위해 `OrderStatus` (`PENDING`, `ACCEPTED`, `REJECTED`, `CANCELED`)를 도입합니다.

## 논의 과정 (Discussion)
*   **Developer**: "JPA의 `@OneToMany`를 쓰면 코드가 더 직관적이지 않을까요?"
*   **Architect**: "객체 간의 강한 결합은 대규모 트래픽 환경에서 성능 최적화(N+1 문제 등)와 데이터 로딩 제어를 어렵게 만듭니다. ID 기반 참조를 통해 각 엔티티의 경계를 명확히 하고 필요할 때만 조회하는 방식이 확장성에 더 유리합니다."
*   **Developer**: "한 번에 여러 메뉴를 주문할 때 포인트 차감은 어떻게 하나요?"
*   **Architect**: "모든 메뉴의 가격과 수량을 합산한 `totalPrice`를 먼저 계산한 뒤, `PointService`를 통해 단 한 번의 원자적 트랜잭션으로 차감합니다."

## 결과 (Consequences)
- **장점**: 실제 서비스와 유사한 다중 메뉴 주문 기능 제공. 데이터 모델의 확장성 확보.
- **단점**: 주문 조회 시 `OrderItem`을 별도로 조회해야 하는 로직상의 복잡도 소폭 증가.
