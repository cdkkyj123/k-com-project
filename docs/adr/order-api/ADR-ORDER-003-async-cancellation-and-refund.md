# ADR-ORDER-003: 매장 주문 거절 시 비동기 환불(보상 트랜잭션) 전략

## 상태
승인됨 (Accepted)

## 맥락 (Context)
주문이 중앙 서버에서 성공적으로 수수되었더라도, 매장(POS)의 상황(재고 부족, 기기 고장 등)에 따라 주문이 거절될 수 있습니다. 이때 이미 차감된 포인트는 반드시 사용자에게 환불되어야 합니다. 전국 규모의 시스템에서 중앙 서버와 수많은 POS 간의 강한 결합을 피하기 위해, Kafka 이벤트를 통한 비동기 보상 트랜잭션 설계가 필요합니다.

## 결정 (Decision)
1. **이벤트 기반 보상 트랜잭션 (Saga Pattern)**:
    - 매장 POS가 주문 거절 시 `coffee-order-reject` 토픽으로 주문 ID가 포함된 메시지를 발행합니다.
    - 중앙 서버의 `OrderRejectConsumer`가 해당 메시지를 수신하여 비동기적으로 환불 로직을 트리거합니다.
2. **멱등성 보장 환불**:
    - 환불 처리 시 `REFUND-{orderId}` 형식의 멱등성 키를 생성하여 `PointService`를 호출합니다.
    - 이를 통해 동일한 거절 메시지가 중복 수신되더라도 포인트가 중복 환불되는 사고를 원천 차단합니다.
3. **주문 상태 업데이트**:
    - 환불 완료와 동시에 `Order`의 상태를 `REJECTED`로 변경하여 최종 정합성을 맞춥니다.

## 논의 과정 (Discussion)
*   **Developer**: "사용자 앱에서 즉시 환불 결과를 확인할 수 있어야 하지 않나요?"
*   **Architect**: "POS의 거절은 물리적으로 떨어진 매장에서 발생하므로 동기식(HTTP) 처리는 불가능합니다. 앱에서는 '주문 대기' 상태를 보여주고, 환불 완료 시 푸시 알림이나 상태 변경을 통해 사용자에게 알리는 것이 분산 시스템의 표준 방식입니다."
*   **Developer**: "환불 로직 실행 중 장애가 발생하면 어떻게 하나요?"
*   **Architect**: "Kafka의 재시도 메커니즘과 멱등성 키 기반의 `PointService` 덕분에 안전하게 재시도할 수 있습니다. 또한 `ChargeHistory`에 환불 기록이 남으므로 추후 추적도 용이합니다."

## 결과 (Consequences)
- **장점**: 매장과 중앙 서버 간의 느슨한 결합(Loose Coupling) 유지. 장애 상황에서의 데이터 무결성 보장.
- **단점**: 실시간 환불이 아닌 지연 환불(Eventual Consistency) 구조이므로 사용자 UX 설계 시 이를 고려해야 함.
