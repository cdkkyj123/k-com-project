# ADR-STORE-001: 매장 목록 조회 고도화 (페이징, 검색 및 캐싱)

## 상태
승인됨 (Accepted)

## 맥락 (Context)
전국 수천 개의 매장을 관리하는 스마트 오더 시스템의 특성상, 단순히 모든 매장 정보를 한 번에 반환하는 것은 심각한 성능 저하와 리소스 낭비를 초래함. 또한 사용자가 원하는 매장을 신속하게 찾을 수 있는 검색 기능과 빈번한 조회에 대응하기 위한 고성능 캐싱 전략이 요구됨.

## 논의 과정 및 의사결정 이력

### 1단계: 매장 목록 페이징의 필연성 확인
- **의문 (User)**: "매장 찾는 건 왜 페이지네이션도 안 해?"
- **판단 (Developer)**: 대규모 프랜차이즈 환경에서 수천 건의 매장 데이터를 무분별하게 로딩하는 것은 네트워크 및 DB 부하의 원인이 됨을 인정함.
- **결정**: 메뉴 API와 동일한 **Cursor 기반(No-Offset) 페이징**을 적용하여 일관된 고성능 조회 아키텍처를 구축하기로 함.

### 2단계: QueryDSL 검색 기능 도입
- **제안**: 단순히 목록만 보는 것이 아니라, 매장명을 키워드로 검색할 수 있는 기능을 추가 제안.
- **결정**: QueryDSL을 활용해 타입 안정성이 보장된 동적 필터링 로직을 구현하여 사용자 편의성을 증대함.

### 3단계: 매장 조회 캐싱 전략 수립
- **의문 (User)**: "매장 조회 자체도 캐싱할 여지가 있지 않을까?"
- **분석 (Developer)**: 매장 데이터는 '읽기 비중이 압도적으로 높고 변경 주기가 길다'는 특징이 있어 캐싱의 효율이 매우 높음.
- **결정**: Redis 기반의 Cache-Aside 패턴을 적용하고, 검색 키워드와 페이징 위치를 조합한 복합 캐시 키를 운용하기로 함.

---

## 결정 (Decision)
1.  **조회 방식**: `lastId` 기반의 No-Offset 페이징과 QueryDSL 기반 매장명 검색을 통합 구현함.
2.  **캐싱**: `StoreService.getStores` 결과에 `@Cacheable`을 적용하여 Redis에 저장함. (캐시 키 예: `storeList::강남_100_10`)
3.  **최적화**: 결과가 비어있는 경우에는 캐싱하지 않도록 `unless` 조건을 추가하여 Redis 메모리를 효율적으로 사용함.

## 결과 (Consequences)
- **장점**: 
    - 수천 개의 매장이 있어도 인덱스 기반 탐색으로 O(1)에 가까운 조회 성능 보장.
    - 반복적인 매장 검색 및 목록 조회 요청을 Redis에서 처리하여 DB 부하를 획기적으로 감소시킴.
    - 사용자가 원하는 매장을 '이름'으로 빠르게 찾을 수 있는 검색 편의성 확보.
- **단점**: 매장 정보 수정 시 캐시 무효화(Evict) 로직 관리가 필요함.
