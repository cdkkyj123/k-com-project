# [STRESS-TEST-005] 아웃박스 탄력성 검증: 인프라 장애 시 메시지 보존 및 복구

## 1. 테스트 개요
- **테스트 명**: Kafka 브로커 장애 시 Transactional Outbox 패턴의 신뢰성 및 복구 테스트
- **상태**: `PHASE-1: DESIGN`
- **최종 업데이트**: 2026-05-08

---

## 2. 테스트 수행 근거 (Why?)
마이크로서비스 아키텍처에서 DB 트랜잭션과 외부 메시징 발행(Kafka)은 하나의 원자적 트랜잭션으로 묶일 수 없습니다. 따라서 **Transactional Outbox 패턴**을 사용해 먼저 DB에 메시지를 저장합니다. 
실제 인프라 운영 중 Kafka가 일시적으로 중단되거나 지연될 때, 시스템이 주문을 거절하지 않고 정상적으로 처리한 뒤 나중에 메시지를 재발행하는지(Eventual Consistency) 확인해야 합니다.

---

## 3. 테스트 시나리오 및 설정
### 3.1 상황 설정
1. **장애 발생**: Kafka 발행 로직(Producer)에 의도적으로 예외(Timeout 또는 Connection Refused)를 주입하거나 Mocking하여 장애 상황을 시뮬레이션합니다.
2. **주문 실행**: 사용자가 주문을 요청합니다.
3. **상태 확인**: DB의 주문 데이터는 저장되지만, `Outbox` 레코드는 `INIT` 상태에 머물러 있는지 확인합니다.
4. **장애 복구**: Kafka 기능을 정상화합니다.
5. **복구 실행**: `OutboxScheduler`가 주기적으로 돌며 `INIT` 상태의 메시지를 Kafka로 재전송하는지 확인합니다.

### 3.2 핵심 검증 지점
- **비즈니스 연속성**: Kafka 장애와 무관하게 주문은 성공하는가?
- **메시지 무결성**: 장애 복구 후 누락된 메시지가 정확히 Kafka로 발행되어 `PUBLISHED` 상태로 전이되는가?

---

## 4. 예상 결과 (Expectations)
- **가설**: Kafka가 죽어있더라도 주문 트랜잭션은 성공적으로 종료될 것이며, 메시지는 DB의 `Outbox` 테이블에 안전하게 보관될 것이다. 이후 인프라가 정상화되면 스케줄러에 의해 단 한 건의 유실도 없이 최종적으로 Kafka에 도달할 것이다.

## 6. 테스트 결과 및 분석 (Actual Results)
- **상태**: `SUCCESS (PASS)`
- **수행 일시**: 2026-05-08
- **주요 발견**: Kafka 장애 상황에서 메시지 상태가 `FAILED`로 안전하게 전이되었으며, 인프라 복구 후 `SENT` 상태로 최종 발행 성공함.

### 6.1 기술적 보완 사항 (Refactoring)
- **트랜잭션 분리**: 기존 `OutboxScheduler` 내부에서 `processRecord`를 호출할 때 프록시 미적용으로 인해 트랜잭션 롤백 및 상태 업데이트 실패 현상이 발견됨. 이를 `OutboxService`로 분리하여 각 레코드당 개별 트랜잭션을 보장하도록 개선함.
- **최종 일관성**: 일시적인 Kafka 지연이나 중단이 발생하더라도, DB의 아웃박스 데이터는 보존되며 스케줄러를 통해 누락 없는 메시지 전송이 가능함을 입증함.

---

## 7. 결론
Transactional Outbox 패턴은 분산 인프라의 불안정성 속에서도 비즈니스 데이터의 **최종 일관성(Eventual Consistency)**을 보장하는 가장 탄탄한 설계임을 확인함.
