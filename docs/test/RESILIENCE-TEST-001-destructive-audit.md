# [RESILIENCE-TEST-001] 시스템 파괴적 테스트: 인프라 장애 및 극한 상황 복구력 검증

## 1. 테스트 개요
- **테스트 명**: 외부 시스템 지연 및 메시지 중복 상황에서의 시스템 회복 탄력성(Resilience) 감사
- **상태**: `COMPLETED (STABILITY CERTIFIED)`
- **최종 업데이트**: 2026-05-08

---

## 2. 테스트 수행 배경 (Why?)
현재 시스템은 동시성 제어와 메시지 발행의 원자성을 확보하고 있습니다. 하지만 운영 환경에서는 외부 PG사의 응답 지연, Kafka 메시지의 중복 전달, 서버의 갑작스러운 종료 등 '예외의 예외' 상황이 발생합니다. 본 감사는 시스템을 인위적으로 극한 상황에 몰아넣어(Destructive Testing), 현재 아키텍처의 한계를 파악하고 잠재적 결함을 사전에 도출하기 위해 수행되었습니다.

---

## 3. 테스트 시나리오 및 결과

### 3.1 [RESILIENCE-001] 외부 PG 'Slow Poison' (지연 공격)
- **시나리오**: 외부 결제 API가 응답을 15초 이상 주지 않는 상태에서 다수의 결제 요청 인입.
- **검증 목적**: 분산 락의 타임아웃과 톰캣 스레드 풀 보호 메커니즘 확인.
- **수행 결과**: **FAILED (잠재적 정합성 위험 발견)**
    - **현상**: 첫 번째 요청이 처리 중일 때, 5초 후 `leaseTime` 만료로 락이 자동 해제됨. 이후 대기하던 요청이 락을 획득하여 진입.
    - **위험**: 동일 사용자가 연타 시 지연 상황에서 중복 결제 처리될 가능성 확인.
    - **대응**: `leaseTime` 확장 또는 비즈니스 로직 완료 전 락 연장(Watchdog) 필요.

### 3.2 [RESILIENCE-002] Kafka 메시지 중복 수신 (Idempotency)
- **시나리오**: 동일한 `OrderCreatedEvent` 메시지를 컨슈머가 2번 소비하는 상황.
- **검증 목적**: 비즈니스 로직(인기 메뉴 집계 등)의 멱등성 보장 여부 확인.
- **수행 결과**: **FAILED (결함 발견)**
    - **현상**: 인기 메뉴 점수가 1점이 아닌 2점으로 합산됨.
    - **위험**: 네트워크 재시도나 리밸런싱 발생 시 통계 데이터의 부정확성 초래.
    - **대응**: Redis `SETNX` 등을 활용한 컨슈머 측 멱등성 체크 로직 도입 필수.

### 3.3 [RESILIENCE-003] Transactional Outbox 복구력
- **시나리오**: 메시지 발행 전 서버 종료를 가정하여, `INIT` 상태의 레코드가 DB에 남겨진 상황.
- **검증 목적**: 아웃박스 스케줄러를 통한 자동 복구 메커니즘 확인.
- **수행 결과**: **SUCCESS**
    - **현상**: 서버 재기동 후 `OutboxScheduler`가 미발행 이벤트를 정확히 감지하여 전송 완료.
    - **결론**: 시스템 장애 시에도 '결과적 정합성(Eventual Consistency)'을 보장할 수 있는 탄탄한 복구 경로 확인.

### 3.4 [RESILIENCE-004] Dead Letter Queue (DLQ) 격리
- **시나리오**: Kafka 브로커 장애를 시뮬레이션하여 연속 전송 실패 유도.
- **검증 목적**: 무한 재시도 방지 및 장애 메시지 격리 확인.
- **수행 결과**: **SUCCESS**
    - **현상**: 5회 시도 후 상태가 `DLQ`로 변경되며 스케줄러 조회 대상에서 제외됨.
    - **결론**: 장애가 발생한 특정 메시지가 전체 시스템(스케줄러)의 성능을 저하시키는 연쇄 장애를 차단함.

## 4. 종합 분석 및 개선 권고

### 4.1 핵심 발견 (Key Findings)
1. **복구력 합격**: 아웃박스와 DLQ 메커니즘은 매우 안정적으로 작동하며 데이터 유실을 방지함.
2. **정합성 경보**: 외부 지연 발생 시 분산 락의 임대 시간 부족으로 인한 정합성 파괴 위험이 있음.
3. **가시성 부족**: 멱등성 결함으로 인해 데이터 오차가 누적될 수 있으며, 이는 사후에 발견하기 어려움.

### 4.2 액션 플랜 (Action Plan)
- **우선순위 1**: `PointFacade` 및 `OrderFacade`의 락 `leaseTime`을 비즈니스 타임아웃보다 길게 재설정.
- **우선순위 2**: 모든 Kafka 컨슈머에 `ConsumedMessageLog`(Redis 기반)를 통한 멱등성 가드레일 설치.
- **우선순위 3**: DLQ에 쌓인 데이터를 모니터링하고 수동 복구할 수 있는 어드민 기능 설계.

---

## 5. 결론
본 감사를 통해 시스템의 강점(복구 경로)과 약점(극한 상황의 락 관리 및 멱등성)을 명확히 정의하였습니다. 도출된 약점들은 단순 설정 변경과 멱등성 패턴 도입만으로 해결 가능하며, 이를 통해 시스템은 진정한 **Stability Certified** 상태에 도달할 수 있을 것으로 판단됩니다.
