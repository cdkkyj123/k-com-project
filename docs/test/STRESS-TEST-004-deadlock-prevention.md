# [STRESS-TEST-004] 데드락 방지 검증: 다중 락 획득 순서 보장 테스트

## 1. 테스트 개요
- **테스트 명**: 역순 다중 메뉴 주문 시 데드락(Deadlock) 발생 여부 검증
- **상태**: `PHASE-1: DESIGN`
- **최종 업데이트**: 2026-05-08

---

## 2. 테스트 수행 근거 (Why?)
여러 개의 자원(메뉴 재고)에 대해 락을 획득할 때, 스레드마다 락을 잡는 순서가 다르면 서로 상대방의 자원을 기다리며 무한 대기에 빠지는 **데드락**이 발생할 수 있습니다. 
현재 `OrderFacade`에는 이를 방지하기 위해 `menuId` 기준 오름차순 정렬 로직이 포함되어 있습니다. 이 로직이 실제 고부하 환경에서 데드락을 완벽히 차단하는지 입증해야 합니다.

---

## 3. 테스트 시나리오 및 설정
### 3.1 상황 설정
- **자원**: 메뉴 A (ID: 1), 메뉴 B (ID: 2)
- **사용자**: 
    - 그룹 1 (50명): [메뉴 1, 메뉴 2] 순서로 주문 요청.
    - 그룹 2 (50명): [메뉴 2, 메뉴 1] 순서로 주문 요청 (의도적 역순).
- **반복 횟수**: 총 100건의 동시 요청을 수차례 반복 실행.

### 3.2 핵심 검증 지점
- **시스템 가용성**: 모든 요청이 타임아웃 없이 성공하거나, 재고 부족 시 정상적으로 실패 응답을 반환하는가? (시스템이 멈추지 않는가?)
- **정렬 유효성**: 로그를 통해 실제 Redis 락 획득 시도가 메뉴 ID 순서대로 이루어지는지 확인.

---

## 4. 예상 결과 (Expectations)
- **가설**: `OrderFacade`의 정렬 로직 덕분에 그룹 2의 역순 요청도 실제로는 [메뉴 1, 메뉴 2] 순서로 락을 요청하게 된다. 따라서 락 획득 순서가 일치되어 데드락 없이 모든 트랜잭션이 순차 처리되거나 안전하게 롤백될 것이다.

## 6. 테스트 결과 및 분석 (Actual Results)
- **상태**: `SUCCESS (PASS)`
- **수행 일시**: 2026-05-08
- **최종 지표**: 100건의 동시 역순 주문 중 100건 모두 성공 (데드락 발생 0건)

### 6.1 성공 요인 분석
- `OrderFacade.java`에서 `itemRequests`를 `menuId` 기준으로 정렬한 후 락을 획득함으로써, 모든 스레드가 동일한 순서(A -> B)로 자원을 요청함.
- 이로 인해 원형 대기(Circular Wait) 조건이 원천 차단되어 대규모 부하 상황에서도 시스템 멈춤 현상 없이 안전하게 처리됨을 확인.

---

## 7. 결론
다중 자원 락 획득 시 **정렬(Sorting)** 전략은 데드락을 방지하는 가장 확실한 방법임을 입증함.
