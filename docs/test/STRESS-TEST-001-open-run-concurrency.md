# [STRESS-TEST-001] 오픈런 시나리오: 재고 및 포인트 동시성 정합성 검증

## 1. 테스트 개요
- **테스트 명**: 한정 수량 메뉴에 대한 대규모 동시 주문(오픈런) 시뮬레이션
- **상태**: `PHASE-1: DESIGN & EXPECTATION` (테스트 실행 전 설계 단계)
- **최종 업데이트**: 2026-05-07

---

## 2. 테스트 수행 근거 및 배경 (Why?)
### 2.1 분산 환경의 정합성 검증 필요성
현재 시스템은 **Redisson 분산 락**을 사용하여 포인트와 재고를 보호하고 있습니다. 이론적으로는 정합성이 보장되어야 하지만, 실제 네트워크 지연, Redis 부하, DB 트랜잭션 격리 수준 등이 복합적으로 작용할 때 예기치 못한 '더블 차감'이나 '재고 초과 판매'가 발생할 수 있습니다.

### 2.2 비즈니스 시나리오
인기 있는 카페에서 특정 시간에 한정판 굿즈나 메뉴를 판매할 때, 수백 명의 사용자가 동시에 '주문하기' 버튼을 누르는 상황을 가정합니다. 이 상황은 시스템에서 가장 높은 부하가 발생하는 지점이며, 데이터 무결성이 깨질 경우 금전적 손실과 고객 신뢰 하락으로 직결됩니다.

---

## 3. 테스트 파라미터 및 설정 근거
### 3.1 동시 접속자 수: 100명
- **선정 근거**: 단일 매장 기준으로 초당 100건의 주문 요청은 '극심한 혼잡' 상태를 대변합니다. 일반적인 WAS(Tomcat) 스레드 풀이 200개 내외임을 고려할 때, 100개의 스레드가 동시에 락을 경쟁하는 상황은 자원 경합(Contention)을 확인하기에 적절한 수치입니다.
- **확장 계획**: 100명 테스트 성공 시, 500명까지 순차적으로 증가시켜 시스템의 임계치(Break-point)를 확인합니다.

### 3.2 재고 수량: 10개
- **선정 근거**: 요청 수(100)보다 훨씬 적은 자원(10)을 설정함으로써, **90%의 요청이 실패**해야 하는 상황을 만듭니다. 이는 단순 성공 테스트가 아니라 '정교한 실패 처리'와 '락 획득 실패 시의 안정성'을 검증하기 위함입니다.

### 3.3 타임아웃 설정 (Wait Time: 5s, Lease Time: 3s)
- **선정 근거**: ADR-008(인프라 타임아웃 전략)에 따라, 사용자가 최대 5초까지는 대기할 수 있다고 가정합니다. 락 점유 시간은 비즈니스 로직(재고 확인 + 포인트 차감 + 주문 저장)을 충분히 처리할 수 있는 3초로 설정합니다.

---

## 4. 테스트 시도 목적 (Objectives)
1. **재고 무결성**: 100번의 요청 중 정확히 10건만 주문이 생성되고, `MenuStock`의 잔여 수량이 0이어야 함.
2. **금전 무결성**: 성공한 10명의 포인트만 차감되어야 하며, 나머지 90명은 락 획득 실패 혹은 재고 부족으로 인해 포인트 변동이 전혀 없어야 함.
3. **데드락 방지**: 메뉴 ID 정렬 로직이 멀티스레드 환경에서 데드락 없이 순차적으로 락을 획득/해제하는지 확인.
4. **성능 지표**: 100개의 요청이 모두 처리될 때까지의 전체 소요 시간 및 평균 응답 시간 측정.

---

## 5. 테스트 수행 과정 (Methodology)
1. **사전 준비**:
    - 테스트용 사용자 100명 생성 및 각 10,000 포인트 충전.
    - 테스트용 매장 및 수량이 10개인 한정 메뉴 생성.
2. **동시성 제어**:
    - `CountDownLatch`를 사용하여 100개의 스레드를 대기시킨 후, 동시에 `executeOrder` 호출.
3. **결과 수집**:
    - 성공/실패 횟수 카운트.
    - DB 내 `Order`, `OrderItem`, `User`, `MenuStock` 상태 최종 확인.
4. **예외 상황 분석**:
    - `REDISSON_LOCK_FAILED`, `OUT_OF_STOCK` 등 예외 로그 분석.

---

## 6. 예상 결과 및 가설 (Expectations)
- **가설 1**: Redisson의 Pub/Sub 기반 스핀 락 구조 덕분에 CPU 부하를 최소화하면서도 10건의 주문이 정확히 처리될 것이다.
- **가설 2**: 락 획득 대기 시간(5s) 내에 처지되지 못한 일부 요청은 `POINT_LOCK_FAILED` 혹은 `STOCK_LOCK_FAILED` 예외를 던지며 안전하게 롤백될 것이다.
- **가설 3**: DB 트랜잭션이 락 범위 내에 포함되어 있으므로, 락 반납 전 커밋이 완료되어 정합성 이슈(Dirty Read 등)는 발생하지 않을 것이다.

- `docker-compose`를 통한 외부 인프라 의존성이 해결되지 않은 상태에서 테스트가 수행됨.

## 9. 예상과 실제 결과 비교 및 보완점 (Analysis)
- **예상**: Redisson 락이 정상 작동하여 10건의 정합성을 보장할 것으로 기대함.
- **실제**: 인프라 의존성으로 인해 비즈니스 로직 검증 단계까지 진입하지 못함.
- **보완점**: 
    1. **Embedded Redis 도입**: 외부 Redis 없이도 테스트가 가능하도록 `it.ozimov:embedded-redis` 등의 라이브러리 추가 검토.
    2. **Testcontainers 활용**: 도커 환경이 보장되는 곳에서만 테스트가 수행되도록 `@Testcontainers` 설정 강화 (현재 환경은 도커 미지원 확인).
    3. **Mocking 전략**: `RedissonClient`를 Mocking하여 로직만 검증할 것인지, 아니면 실제 락 메커니즘을 검증할 것인지 결정 필요 (본 테스트는 '심화 테스트'이므로 실제 락 검증이 가치 있음).

---

## 10. 결론 및 차기 단계
현재 환경에서는 분산 락을 포함한 심화 테스트 수행이 불가능하므로, **테스트 전용 인메모리 Redis 환경**을 구축한 후 재시도가 필요합니다.
